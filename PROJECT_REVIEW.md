# Независимая оценка проекта AusTender Data Platform

Дата: 2026-07-17. Оценка по коду в репозитории (модели dbt, SQL-скрипты, ingestion, CI, README, история коммитов), без запуска на живом аккаунте.

## Резюме

Проект существенно выше уровня типичного пет-проекта. Все четыре требования Medavie закрыты предметно, а не декоративно: Medallion — с осмысленным выбором грануляции на каждом слое; RBAC — с ownership chaining и проверкой через secondary roles; dbt — с инкрементальным фактом, снапшотом SCD2, кастомными тестами, которые охраняют реальные регрессии, а не наличие колонок. Главный актив — инженерный нарратив: история коммитов с попыткой ключевания по ABN, ревертом и последующим решением через реестр читается как настоящий продакшн-процесс.

Главные слабости: (1) изоляция CI держится на макросе, а не на грантах — роль CI фактически может переписать прод; (2) заявлен CI/CD, но CD нет — прод собирается вручную; (3) 26.6M строк ABR пересобираются каждым `dbt build`, включая каждый PR.

## Часть 1. Взгляд архитектора

### Что сделано сильно

- **Выбор грануляции обоснован и охраняется тестами.** Silver хранит каждое notice, Gold схлопывает цепочку поправок до актуальной версии, и `assert_gold_matches_silver` независимо перевыводит ожидаемого «победителя» — это внешний контроль инкрементального merge, а не пересказ его логики.
- **Инкрементальный факт написан правильно.** Watermark по `loaded_at`, а не по бизнес-дате; ранжирование против всей цепочки, а не только новых строк; `chain_loaded_at` двигает watermark даже когда поздняя поправка проигрывает. Это три классические ошибки инкрементальности, и все три явно закрыты.
- **Проблема ненадёжного ABN решена честно.** Вместо эвристики — второй Bronze-источник (реестр как арбитр), `supplier_entity_key` с ABN-ключом только там, где реестр ручается, и name-fallback в «восстановимую» сторону. Ошибочное решение не спрятано, а оставлено в истории коммитов с ревертом.
- **Загрузчик fail-safe.** COPY в scratch-таблицу, атомарная публикация, `ON_ERROR='ABORT_STATEMENT'` — падение загрузки не уничтожает bronze.
- **MART как контракт с потребителем.** Аналитик физически не может просуммировать сырой факт: GOLD не выдан, доступ только через ownership chaining. «Правильный запрос по умолчанию» — редкая зрелость для пет-проекта.

### Находки

**Высокий приоритет**

1. **Роль CI перегрантована — изоляция поведенческая, а не структурная.** В `01_setup_infra_rbac.sql` (п. 3.4) `austender_ci` получает `ALL` на прод-схемы SILVER/GOLD/MART и `ALL ON ALL/FUTURE TABLES IN DATABASE`. Запись в `CI_*` обеспечивает только макрос `generate_schema_name`; PR, меняющий макрос (или баг в нём), собирёт прод-GOLD из ветки. Least privilege: CI-роли достаточно `USAGE` на database, `SELECT` на bronze и `CREATE SCHEMA` — на созданных ею `CI_*` схемах она и так owner.
2. **CI есть, CD нет.** Workflow валидирует PR и push в main, но всегда в `--target ci` (в `CI_*`). Прод-GOLD/MART обновляется вручную с dev-таргета под личным паролем. Для заявки «CI/CD» нужен deploy-job на merge в main (target prod, тем же service-user'ом) — или честная формулировка «CI» в README.
3. **ABR-staging (26.6M строк) пересобирается каждым build.** `stg_abr_names` и `stg_abr_entity` — таблицы, зависящие только от bronze, но они в общем графе: каждый PR копирует 26.6M строк в `CI_SILVER` (время XSMALL-warehouse, кредиты, дубли storage). Решение: тег + `--exclude` в CI, либо отдельный редкий job/incremental.

**Средний приоритет**

4. **CI-запуски конкурируют за одни схемы.** Все PR пишут в одну `CI_SILVER/CI_GOLD`; параллельные PR затирают друг друга, в workflow нет `concurrency`, схемы никогда не чистятся. Минимум — concurrency group; лучше — суффикс с номером PR и drop в конце.
5. **Недетерминированный дедуп в Silver.** Тай-брейк `order by loaded_at desc`, но внутри одной загрузки `_loaded_at` одинаков у всех строк (statement-level `CURRENT_TIMESTAMP`). Сегодня cnid уникален в файле, так что не стреляет, но при интрафайловом дубле победитель случаен. Добавить детерминированный второй ключ сортировки.
6. **`assert_supplier_rollup_never_merges_abns` почти тавтологичен.** ABN-группы содержат один ABN по построению ключа; NAME-группы отсекает `WHERE abr_entity_name is not null and not placeholder`. Как регрессионный гвард на смену логики ключа он полезен, но текущую конструкцию он проверить не может — стоит отметить это в комментарии или усилить (например, проверять по `resolved_abn`).
7. **Обратная сторона name-fallback не проговорена.** NAME-ключ склеивает двух *разных* без-ABN поставщиков с одинаковым нормализованным именем — это over-merge, та самая «невосстановимая» сторона, но в README трейдофф описан только как over-split. Риск мал, но раз уж проект строится на честности про качество данных — стоит одну фразу добавить.
8. **`source freshness` настроен, но нигде не запускается.** CI делает `build` и `docs generate`; шага `dbt source freshness` нет — конфиг мёртвый.
9. **CRLF без `.gitattributes`.** Прямо сейчас 5 файлов «изменены» фантомно (diff пуст при `--ignore-all-space`). Нормализовать (`* text=auto`) и закоммитить.
10. **Password-auth для dev-профиля.** Snowflake сворачивает single-factor password (MFA enforcement); на свежем trial это может просто не завестись. Key-pair уже освоен для CI — логично перевести и человеческий профиль (или задокументировать PAT/MFA).

**Низкий приоритет**

11. Дрейф комментариев: заголовок `01_...sql` говорит «три схемы» (их четыре с MART); комментарий в п. 3.6 про «CI reuses these credentials» противоречит README, где CI уже ходит под `austender_ci_svc`.
12. Seed `known_non_suppliers` без yml: нет `unique` на `normalized_name` — дубль строки в seed размножит строки `rpt_contracts` (поймается unique-тестом на contract_id, но косвенно и с невнятной диагностикой).
13. Аналитик делит `austender_wh` с DE — тяжёлый BI-запрос конкурирует с ETL. Для trial нормально, для «продакшн-паттерна» — отдельный warehouse.
14. `is_weekend` в `dim_date` зависит от сессионного `WEEK_START`. Сегодня дефолт даёт верный результат, но это скрытая зависимость от параметра.
15. `analysis-paths` объявлен, каталога `analyses/` нет — примеры аналитических запросов так и не появились.

## Часть 2. Взгляд пользователей

### Аналитик / BI-потребитель

Сильно: MART делает правильный запрос дефолтным — `is_attributable` с `unattributable_reason`, entity-rollup (Hays — одна строка), обе меры сходятся с фактом до цента (и это охраняется тестом). Колоночные описания в yml — реальная документация, не заглушки.

Чего не хватает:

- **Дат контракта в `rpt_contracts`.** Есть только `publish_date`; `contract_start/end` и `duration_days` до марта не доезжают — первый же вопрос «какие контракты активны сейчас?» отправит аналитика обратно в закрытый GOLD.
- **Отчёта по категориям** (UNSPSC — готовое измерение, отчёта нет) и грануляции мельче года в `rpt_agency_spend`.
- **Опубликованной документации.** `dbt docs generate` в CI выбрасывает результат; аналитик без dbt каталога и lineage не увидит. GitHub Pages решает это одним шагом workflow.
- `rpt_supplier_history` пуст, пока не запущен `dbt snapshot` — для потребителя это выглядит как сломанная вьюха; каveat есть только в yml.

### Нанимающий менеджер / рекрутер (главный пользователь резюме-проекта)

- **README глубокий, но это 15-минутный текст без входа за 60 секунд.** Нет TL;DR, нет CI-бейджа, нет ни одной картинки. Решение о собеседовании принимается быстрее, чем читается раздел про ABN.
- **Диаграмма архитектуры невидима:** `afc.drawio.pdf` попадает под `*.pdf` в .gitignore — на GitHub её нет. Экспортировать в PNG/SVG и вставить в README.
- **Обещанный стрелкой `──▶ Power BI` дашборд отсутствует.** Это последний недостающий артефакт «до витрины»: один скриншот дашборда в README даёт больше, чем ещё тысяча слов.
- Сила, которую стоит подсветить явно: история коммитов (`Key dim_supplier...` → `Revert...` → решение через реестр). Немногие кандидаты показывают, как они ошибаются и чинят; одна строка в README «см. историю коммитов про выбор ключа» превратит это в фичу.

### Инженер-сопровождающий

Onboarding образцовый: `bootstrap.py` идемпотентен, `.env.example`, комментарии объясняют «почему», а не «что». Минусы: грязное рабочее дерево (CRLF-шум) подрывает доверие к репозиторию при клонировании; нет линтера SQL (sqlfluff) и Slim CI (`state:modified` + defer) — на 20 моделях не больно, но как «продакшн-паттерн» просится.

## Топ-5 действий по соотношению эффект/усилие

1. Урезать гранты `austender_ci` до bronze-read + `CREATE SCHEMA` (закрывает главный архитектурный риск, ~10 строк SQL).
2. Добавить deploy-job на merge в main (target prod) — превращает «CI» в честный «CI/CD».
3. Вынести ABR-staging из PR-сборок (`--exclude tag:abr` в CI).
4. README: TL;DR-блок, CI-бейдж, PNG диаграммы, скриншот дашборда (когда появится).
5. `.gitattributes` + закоммитить нормализацию строк; заодно — `dbt source freshness` шагом в CI.
